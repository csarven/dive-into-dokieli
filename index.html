Abstract

In this article we describe dokieli, a general purpose clientside tool for document authoring, editing and publication. Features are enabled progressively according to the needs and technical resources of the user. The editor is open source, built on web standards and documents are represented as linked data. This allows: decentralised storage and data ownership; direct in-browser editing from an LDP-based personal data store; social interactions with documents (such as annotations and replies), and notifications thereof; fine-grained data embedding through RDFa; rich metadata such as provenance and licensing information; enhancements for academic research papers including embedding of datasets and runnable experiments.

Introduction

dokieli is a decentralized article authoring, annotation, and social pingback tool which works from a Web browser. While it is a general purpose tooling to write articles, it is fully compliant with the Linked Research initiative and principles, and provides features and interactions for scholarly communication. Its architecture is progressively enhanced such that dokieli based articles are accessible anywhere from a Line Mode Browser to Firefox Nightly. Articles and annotations can be curled and dereferenced to get the complete content in HTML and RDF.

Where applicable, dokieli can employ participant's e.g., authors, reviewers, commenters, WebID and personal online datastores to store and give different access controls to the information. Similarly, anyone that wants to annotate can store their notes at a Webspace in which they control.

    • Architecture

    Articles are written to share knowledge. The architecture we have chosen enables this knowledge to be as widely accessible as possible; the capabilities of dokieli are designed such that they are enabled progressively and individually, according to the user's needs and technical resources at the time of use. The most minimal tooling required to read content produced with dokieli is a commandline client or text-only browser, and to write content is a text editor. Given a modern web browser, linked data tooling, or an LDP-compatible server, documents and data can be consumed, created, connected and manipulated in more powerful and visually appealing ways.

    HTML+RDFa

    The application of the RDF language is the fundamental choice, and it can be notated in different ways. However, an ongoing challenge is to determine and incorporate appropriate RDF serialization(s) for both data publication and consumption. This is particularly important for different contexts given that each RDF serialization has its own strengths and weakness.

    Why HTML+RDFa: The granularity in which users can mark and create statements for prose content is suitable in HTML+RDFa because out of the box, 1) HTML provides the core language for WWW to represent text-level semantics, sectioning, grouping and embeddable content, tabular data, as well as forms and interactivity to help publishers and Web application developers, and 2) RDFa compliments it by providing the mechanism to express structured data using terms from multiple independently-developed vocabularies to enhance functionality and interoperability of the data. This particular combination makes it possible to have individual documents to be both human and machine-readable.

    RDF in HTML: While transformations between different RDF notations are trivial, our decision to use RDFa in HTML over alternative approaches; Turtle and JSON-LD, is as follows. RDFa can be purposed towards the human-visible (or accessible) portions of the document, and are typically marked inline and in context of where the ideas occur in the document. It is to treat the information available as first-class data as opposed to metadata (which may be hidden from view). While Turtle and JSON-LD may also be written inline, they tend to act as raw data islands, and are slightly disconnected from the context that is human-readable in an HTML document. Concerning data quality, information marked with RDFa is less likely to rot or go stale because of its visibility (or accessibility) to humans, and due to authoring from their canonical location. It should however be noted that Turtle and JSON-LD can be well-purposed in the case of interactive documents. While our justification is aimed at using RDFa for semantic representations, we focus on a user interface that allows the author to work at the level of the RDF language, as opposed to on the syntax level.

    Alternative to HTML+RDFa: It is noteworthy to emphasize on the choice of data representation and its usage, is not done in isolation, and depends on the platforms and its users' needs. In order to contrast HTML+RDFa to the alternatives, we can have a look at one particular option for the purpose of our discussion: RDF Turtle with Markdown to represent the prose content.

    Turtle can be utilized for RDF graphs to be written in a compact and natural text form. Markdown is a plain text formatting syntax designed so that it can be converted to HTML. Hence, this particular combination is suitable for lightweight data representation that is as well human and machine-friendly. While Markdown can be entered manually or using WYSIWYG editors, it is convenient given simplicity for common conventions and expressibility in articles. Here, Turtle acts as the content container. There are however, some limitations to this approach. There is no widely used RDF datatype. There is an IETF draft for mediatype for Markdown (text/markdown), however syntax in one Markdown derivative may be ignored or treated differently in another derivative.

    dokieli HTML documents are stored with references to media stylesheets, and scripts. This is a design choice which is intended to provide a default view and interaction for the article. The defaults are the typical experience when consuming applications do not override with their own or user's preferences, meanwhile being possible to override the defaults based on client's capabilities or preferences.

    For instance, Line Mode Browsers were designed to ensure compatibility with computer terminals have only displayed text. Similarly, some Web browsers have a built-in reader mode feature that strips away presentations and behaviours which are accompanied with the document, and changes the page's text size, contrast and layout for better readability. There are also third-party add-ons which act in a similar manner. As for the structured prose content, it remains possible to transform the RDF triple statements in RDFa into other RDF serializations.

    dokieli takes on a fault-tolerant design approach, i.e., if the scripts or stylseheets are no longer available, the content is still accessible. This allows continuous operation, possibly at a reduced level, rather than failing completely. A single document in HTML+RDFa is able to retain all of its core functionality and content without external dependencies.
